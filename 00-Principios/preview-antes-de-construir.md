# Principio · Preview antes de construir

**Regla:** antes de implementar algo con interfaz o con decisiones de alcance, se muestra un **preview** (mockup, boceto, o spec con ejemplos) y se **aprueba**, antes de escribir el código. Flujo: *mockup → aprobación → spec → implementación.*

**Por qué:** corregir en un mockup cuesta minutos; corregir en código desplegado cuesta horas y reprocesos. El preview alinea expectativas y caza malentendidos temprano.

**Cómo aplicarlo:** para UI, un mockup mobile + desktop antes de codear. Para lógica, una spec con criterios de aceptación y ejemplos. El que diseña no es el que aprueba.
